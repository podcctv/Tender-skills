# Tender-skills

面向投标/售前方案编制的可移植 AI skill。核心 skill 名称为 `tender-bid-writer`，用于让 Codex、OpenClaw、Hermes 等 agent 从招标文件中抽取需求、生成投标方案框架、编写章节、合稿质检、模拟专家评审并输出交付清单。

流程将“需求抽取 -> 章节 brief -> 人工确认 -> 逐章编写 -> 合稿 -> 质检 -> 多专家评审 -> 多轮修订 -> 最终交付”固化为一个可复用的 skill。

      openai.yaml
    references/

- `SKILL.md`：skill 主入口，定义触发场景和执行流程。

- `references/checklists.md`：需求台账、章节 brief、质检、专家评审模板。
- `references/platform-compatibility.md`：Codex、OpenClaw、Hermes 适配说明。
- `scripts/bid_quality_check.py`：可选的本地质检脚本。

## 适用场景

使用这个 skill 处理以下任务：

- 分析招标文件、评分办法、技术规范、商务条款、资格要求。
- 接到外包/代写标书时，先审招标文件，输出对客户的标书框架、资料清单、评分点响应关系和风险反馈。
- 在编写前判断项目属于 `goods-mode`、`software-platform-mode`、`hybrid-goods-software-mode` 或 `service-mode`。
- 将采购需求拆解到三级：一级需求域/系统边界，二级功能或服务事项，三级最小响应单元。
- 生成投标技术方案、服务方案、实施方案、运维方案、培训方案等框架。
- 建立评分点/响应点/证据材料的追踪矩阵。
- 按厚标书标准逐章编写 HTML、Markdown 或 DOCX 目标稿件。
- 合稿后进行占位符、漏项、证据缺口和一致性检查。
- 模拟合规、技术、评分、交付、商务/法务等多专家评审。

## 核心写作标准

- 招标文件是唯一事实源。不得编造资质、证书、案例、人员、价格、工期、授权、签章状态或第三方承诺。
- 外包/代写项目必须先做前置审标。没有拿到客户材料前，只能形成审标报告、框架、资料清单、评分映射和反馈问题，不能把缺失证据写成正式承诺。
- 先判断投标模式，再决定抽取方式、证据矩阵、目录结构、篇幅规划、正文密度和质检规则。
- 采购需求必须先细化到三级。第三级应能直接对应正文小节、响应证据、交付物或验收点。
- 无招标页数硬限制时，按项目价值、项目复杂度和投标模式规划整标篇幅。软件平台标和复杂混合标可用 `1 万元 ≈ 1 页` 作为基础测算；重大软件平台/复杂混合项目原则上不低于约 `300-500 页`。货物标不机械套用长正文，页数主要来自参数表、证据矩阵和附件材料完整度。
- 最小正式正文小节通常不少于 `3-5 段`。货物标参数响应、固定格式表、符合性声明或招标要求短答除外。每段应有机制、场景、执行细节、表单记录、风险控制或验收支撑，不能只换说法重复标题。
- 表格必须服务评审和落地，例如需求响应矩阵、接口联调检查表、数据质量检查表、风险预防表、交付文档清单、验收映射表。
- 正式章节不得出现 `TODO`、`待确认`、`公司名称`、`项目名称：填写`、`如有证据再补` 等内部草稿痕迹。

## 外包标书前置审标流程

当标书是别人外包给你写，或对方只给了招标文件、还没有提供公司资质/业绩/人员/产品/授权/方案素材时，不要直接写正式正文。先用 skill 做一轮“审标 + 框架 + 要材料 + 评分映射 + 对外反馈”。

### 第一轮必须输出

| 输出物 | 用途 |
| --- | --- |
| 招标文件审查报告 | 说明项目概况、投标模式、采购对象、截止时间、资格/格式/签章/否决项风险 |
| 标书整体框架 | 给出目录、章节目标、页数预算、章节和采购需求/评分项的对应关系 |
| 评分点响应映射表 | 把每个评分项映射到响应策略、目标章节、所需证据、当前材料状态和缺失风险 |
| 资料清单 | 告诉客户必须提供哪些材料、用于哪个评分点/条款/章节、优先级和截止时间 |
| 问题澄清清单 | 列出需要客户确认或向采购人澄清的问题 |
| 对外反馈话术 | 可直接发给客户，说明需要什么、为什么需要、不提供会影响什么分数或合规点 |

资料优先级：

- `P0`：资格、否决项、强制格式、授权、签章、承诺函等硬性材料；缺失时不能假定满足。
- `P1`：直接影响评分的关键材料，如业绩、人员证书、产品证明、检测报告、案例、方案素材。
- `P2`：支撑可信度和润色的辅助材料，如公司介绍、获奖、宣传册、一般制度文件。

推荐提示词：

```text
使用 $tender-bid-writer 按外包标书前置审标流程处理这个项目。现在先不要写正式标书正文。请先审招标文件，判断投标模式，输出招标文件审查报告、标书整体框架、评分点响应映射表、完成整套标书需要的资料清单、问题澄清清单，以及一份可以发给客户的反馈说明。资料清单必须说明每项材料对应哪个评分点、招标条款、目标章节、优先级和缺失风险。
```

### 客户回传材料后的第二轮

拿到客户材料后，先核对材料，不要马上合稿。

推荐提示词：

```text
使用 $tender-bid-writer 根据客户新提供的材料更新前置审标结果。请逐项核对资料清单，标注已收到、可用、不完整、过期、与要求不一致或仍缺失；同步更新评分点响应映射表、章节 brief 和可正式写入正文的承诺范围。对仍缺失的材料输出二次追问清单。
```

第二轮确认后，再进入逐章正文编写。

## 四种投标模式

使用这个 skill 时，第一步先让 agent 判定模式。模式会控制后续需求抽取、证据准备、目录框架、篇幅规划和质检重点。

| 模式 | 适用场景 | 核心材料 | 写作厚度 |
| --- | --- | --- | --- |
| `goods-mode` 货物标 | 采购设备、硬件、成品软件、标准产品、耗材、仪器等 | 产品彩页、检测报告、厂家授权、参数响应表、证书、质保、供货安装方案 | 最固定，重点是参数、证据、偏离表，正文不宜虚胖 |
| `software-platform-mode` 软件平台标 | 定制开发、系统集成、数据中台、业务平台、AI 能力中心、软件运维平台等 | 需求分析、总体架构、功能设计、业务流程、数据/接口/安全/实施/验收方案 | 最适合厚标书，按功能、流程、数据对象和场景大篇幅展开 |
| `hybrid-goods-software-mode` 货物+软件混合标 | 既有硬件设备/成品产品，又有平台开发、集成、部署、服务 | 产品参数材料 + 平台方案 + 集成部署 + 联调验收 | 拆成产品响应线、软件方案线、集成交付线，不能混写 |
| `service-mode` 服务标 | 运维、咨询、规划、培训、测评、监理、数据治理、驻场服务等 | 服务范围、人员、SLA/KPI、流程、工具、记录、报告、考核、验收 | 中到厚，重点是人员、流程、记录、考核、风险和验收 |

模式判断推荐提示词：

```text
使用 $tender-bid-writer 先判断本项目属于 goods-mode、software-platform-mode、hybrid-goods-software-mode 还是 service-mode。请说明判断依据，并按对应模式列出需求抽取表、证据清单、目录框架和页数规划。先不要写正文。
```

### goods-mode 货物标

货物标的核心不是长篇架构，而是参数、型号、品牌、证据和偏离控制。

推荐产物：

- 参数响应表
- 产品证据矩阵
- 品牌/型号/规格清单
- 检测报告、彩页、官网截图、证书、授权材料清单
- 偏离表和负偏离风险表
- 供货、安装、调试、培训、验收安排
- 售后服务与质保承诺

推荐提示词：

```text
使用 $tender-bid-writer 按 goods-mode 处理本项目。请优先抽取技术参数、品牌型号、规格要求、证据材料、授权质保、供货安装、调试培训和验收要求，生成参数响应表、产品证据矩阵、偏离风险表和交付清单。不要写成软件平台架构方案。
```

### software-platform-mode 软件平台标

软件平台标适合厚写，要体现业务理解、总体架构、功能模块、数据对象、接口关系、安全、实施、演示、验收和运维。

推荐提示词：

```text
使用 $tender-bid-writer 按 software-platform-mode 处理本项目。请将采购需求拆到三级，按业务场景、功能模块、数据对象、接口关系、安全控制、实施迁移、演示支撑、验收映射和运维服务生成厚标书目录、页数预算和章节 brief。
```

### hybrid-goods-software-mode 混合标

混合标必须拆三条线，避免把硬件参数藏进软件章节，也避免把软件集成写成简单供货。

三条线：

- 货物/产品响应线：型号、品牌、参数、彩页、检测报告、授权、质保、供货。
- 软件/平台方案线：架构、功能、数据、接口、安全、实施、演示、验收。
- 集成交付线：设备安装、软件部署、接口联调、系统测试、培训、试运行、最终验收。

推荐提示词：

```text
使用 $tender-bid-writer 按 hybrid-goods-software-mode 处理本项目。请拆成货物/产品响应线、软件/平台方案线和集成交付线，分别输出需求台账、证据矩阵、目录框架、责任矩阵、联调验收计划和页数预算。
```

### service-mode 服务标

服务标不要硬套产品参数表，也不要强行写成平台开发。重点是服务范围、人员组织、SLA、流程记录、考核验收和风险处置。

推荐提示词：

```text
使用 $tender-bid-writer 按 service-mode 处理本项目。请抽取服务范围、人员资质、驻场/远程安排、SLA/KPI、服务流程、响应升级、过程记录、报告节奏、考核办法、培训交接和验收要求，生成服务方案框架和证据清单。
```

## 模式设计依据

四模式规则参考了政府采购制度中的几个基础口径：

- [《政府采购需求管理办法》](https://m.mof.gov.cn/czxw/202105/t20210510_3699403.htm)：采购需求包括采购标的及其技术、商务要求，技术要求覆盖功能和质量，商务要求覆盖交付、付款、包装运输、售后服务等。
- [《政府采购货物和服务招标投标管理办法》财政部令第 87 号](https://www.mof.gov.cn/gp/xxgkml/tfs/201707/t20170718_2652766.htm)：评审因素应与货物服务质量相关，包括投标报价、技术或服务水平、履约能力、售后服务等，并应细化量化、对应采购需求和商务条件。
- [《中华人民共和国政府采购法实施条例》](https://www.mof.gov.cn/zhengwuxinxi/zhengcefabu/201502/t20150227_1195516.htm)：政府采购对象和工程相关货物、服务有不同适用边界，工程建设相关内容需要单独识别，避免把施工组织设计类项目混入本 skill 的软件/货物模式。

## 分阶段使用指南

不要一上来要求 agent 直接写完整标书。推荐按阶段推进，每个阶段都留下可追溯产物。

### 1. 资料接收、外包审标、模式判断与工作区整理

目标：确认招标文件、附件、图纸、澄清文件、响应格式、模板和证据材料是否齐全；如果是外包/代写项目，先输出给客户的审标反馈包；并先判定投标模式。

推荐提示词：

```text
使用 $tender-bid-writer 整理当前工作区的招标资料。如果这是外包/代写项目，请先按前置审标流程输出招标文件审查报告、标书整体框架、评分点响应映射表、资料清单和对客户反馈话术。请识别招标文件、附件、澄清文件、格式模板和证据材料，判断本项目属于 goods-mode、software-platform-mode、hybrid-goods-software-mode 还是 service-mode，说明依据，建立 00_source 到 07_client_feedback 的工作目录，并列出缺失或无法读取的文件。
```

阶段产物：资料清单、投标模式判断表、招标文件审查报告、评分点响应映射表、客户材料清单、问题澄清清单、对外反馈话术、工作目录。

### 2. 需求抽取与三级细化

目标：先抽取评分标准、资格要求、技术参数、采购需求、商务条款和格式要求，再把采购需求细化到三级。

三级拆解建议：

| 层级 | 含义 | 示例 |
| --- | --- | --- |
| 一级 | 需求域、系统边界、平台或服务大类 | 数据中台、交易服务平台、运维服务 |
| 二级 | 功能、服务事项、控制对象或交付物组 | 数据治理、接口联调、故障响应 |
| 三级 | 最小响应单元，可写正文、可检查、可验收 | 主数据编码规则、接口异常重试机制、P1 故障处置记录 |

通用推荐提示词：

```text
使用 $tender-bid-writer 按已确认的投标模式抽取招标文件中的评分标准、资格要求、技术参数、采购需求和商务条款。采购需求必须细化到三级，并为每个三级需求标注来源页码/条款、响应章节、证据材料和验收关联。
```

阶段产物：评分台账、资格台账、技术参数台账、商务条款台账、采购需求三级分解表、证据矩阵。

### 3. 整体篇幅规划与章节 brief

目标：在写正文前先确定整标厚度、章节结构、每章目标、证据边界和验收输出。

规则：

- 若招标文件有页数上限、格式模板或固定响应表，优先服从招标文件。
- 若无硬限制，按投标模式规划整体篇幅。软件平台标和复杂混合标可按 `1 万元 ≈ 1 页` 并结合复杂度向上调整。
- 重大软件平台/复杂混合项目整套技术/商务响应原则上不低于约 `300-500 页`。
- 货物标不追求虚胖正文，重点保证参数响应、证据矩阵、偏离表和附件材料完整。
- 服务标按服务范围、人员投入、SLA、考核和记录要求确定厚度。
- 每个三级采购需求应有一个主响应位置，不能散落在多个章节无人负责。

推荐提示词：

```text
使用 $tender-bid-writer 基于需求台账生成投标文件整体框架和章节 brief。请给出整标页数预算、每章目标页数、三级采购需求映射、拟使用表格、证据需求和验收支撑材料。先不要写正文，等我确认框架。
```

阶段产物：目录框架、页数预算表、章节 brief、需求-章节映射表、证据需求清单。

### 4. 逐章厚标书正文编写

目标：按确认后的 brief 一章一章写，不写泛化模板。

正文要求：

- 每个最小正式正文小节通常至少 `3-5 段`，除非是货物标参数响应、固定格式表、符合性声明或招标文件要求短答。
- 优先使用 `机制 + 场景 + 表单 + 输出成果` 结构。
- 平台型软件项目要写到系统边界、数据链路、接口联调、数据迁移、主数据、质量追溯、业务连续性、演示功能等专项。
- 表格要能支撑评分、实施或验收，不能只是装饰。

推荐提示词：

```text
使用 $tender-bid-writer 按已确认的章节 brief 编写《质量管理方案》正文。请覆盖对应三级采购需求，每个最小小节不少于 3-5 段，并加入质量指标表、阶段门禁表、问题分级表、验收映射表和过程记录说明。
```

阶段产物：单章正文、章节表格、过程记录模板、验收映射材料。

### 5. 章节自检与补强

目标：每章完成后先评审，不要直接合稿。

检查重点：

- 是否覆盖评分项、强制要求和三级采购需求。
- 投标模式是否正确，是否存在货物标被写成平台方案、平台标被写成参数表、混合标没有拆线、服务标硬套产品/软件模板等问题。
- 是否存在无证据承诺、草稿痕迹或虚构材料。
- 是否有足够项目场景，而不是通用软件工程模板。
- 每个最小正文小节是否达到 `3-5 段` 且内容实。
- 是否形成可验收、可追踪的输出物。

推荐提示词：

```text
使用 $tender-bid-writer 对刚完成的章节做章级评审。请检查评分覆盖、三级采购需求覆盖、3-5 段正文要求、证据缺口、草稿痕迹、跨章节冲突和验收追踪关系，并给出修订清单。
```

阶段产物：章级评审表、修订清单、证据缺口清单。

### 6. 合稿质检与专家评审

目标：合并章节后，从合规、技术、评分、交付、商务/法务角度模拟评审。

推荐提示词：

```text
使用 $tender-bid-writer 合并已确认章节并运行质检。请输出漏项、短章节、占位符、证据缺口、三级需求未响应项、页数不足项和跨章节冲突，再模拟五类专家评审并给出提分修订计划。
```

阶段产物：合稿文件、QC 报告、专家评审意见、提分修订计划。

### 7. 最终交付

目标：形成提交前清单，确认签章、附件、格式和未决事项。

推荐提示词：

```text
使用 $tender-bid-writer 准备最终交付清单。请核对最终文件、附件、响应表、证据材料、签章要求、格式要求、未决确认事项和提交风险。
```

阶段产物：最终交付清单、签章清单、附件清单、未决风险清单。

## 本地质检脚本

`scripts/bid_quality_check.py` 是一个无第三方依赖的辅助脚本，可以扫描草稿中的明显风险：

- `TODO`、`TBD`、`待补充`、`待确认` 等占位符。
- 过短章节。
- 需求文件和方案正文之间的覆盖缺口。
- 证书、资质、授权、案例、人员等证据引用不足。

示例：

```bash
python skills/tender-bid-writer/scripts/bid_quality_check.py \
  --workspace ./my-bid-workspace \
  --proposal ./my-bid-workspace/03_chapters \
  --requirements ./my-bid-workspace/01_requirements \
  --out ./my-bid-workspace/05_qc/qc_report.md
```

脚本只做机器粗筛，不能替代人工复核。正式投标前必须人工核对否决项、评分标准、格式要求、签章要求和所有证据材料。

## 安全边界

这个 skill 明确要求：

- 以招标文件为唯一事实源。
- 不编造资质、证书、案例、价格、工期、授权、签章状态或第三方承诺。
- 对缺失信息标记 `待确认`。
- 涉及否决项、资格风险、法律/商务承诺时先要求人工确认。
